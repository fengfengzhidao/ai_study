# 怎么理解 Agent 开发的基本思路

## 这节课的目标

学会先理解

Agent 开发到底在做什么

以及一个最基础的 Agent

通常是怎么被搭出来的

很多人一听 Agent 开发

就会马上想到

- 很复杂的框架
- 自动跑很多步的系统
- 能自己完成所有事情的超级助手

这些想象很容易把门槛拉得太高

其实对大多数人来说

一开始更重要的

不是追求一个很强的 Agent

而是先看懂

它和普通对话

普通脚本

普通工作流的区别到底是什么

## 一 Agent 不是会聊天的 AI 而是会推进任务的 AI

普通聊天型 AI

更多是在一问一答

你问一次

它答一次

它当然也可以给建议

但任务通常还是你自己在推进

Agent 的关键差别在于

它不只是回答

而是会围绕一个目标

持续往前推进

比如

- 先理解任务
- 再拆步骤
- 再决定先做哪一步
- 再调用工具
- 再根据结果继续下一步

所以你可以先把 Agent 理解成

带有任务推进能力的 AI

它不是更会聊天

而是更像一个会做事的执行者

## 二 一个最小 Agent 通常包含 4 个部分

如果你把 Agent 开发先讲得简单一点

一个最小 Agent

通常可以先看成 4 个部分

第一

目标

也就是它到底要完成什么任务

第二

上下文

也就是它做事时要参考哪些信息

第三

工具

也就是它能调用什么能力

比如读文件

搜索

运行命令

发消息

第四

循环

也就是它做一步

看结果

再决定下一步

你会发现

这 4 个部分一旦都有了

它就不再只是一次回答

而开始像一个能连着做事的系统

## 三 Agent 开发最容易做错的不是代码 而是边界

很多人一开始做 Agent

最容易关注的是

要不要用某个框架

要不要接很多工具

要不要让它自动做很多步

但真正最容易出问题的

往往不是代码本身

而是边界没有想清楚

比如

- 它到底负责做到哪一步
- 哪些决策必须人来确认
- 哪些工具可以调 哪些不能调
- 失败了以后怎么停下来
- 做完以后怎么把结果交回来

如果这些边界不清楚

就算工具接得很多

Agent 也很容易失控

所以开始做 Agent

先别急着做大

先做一个目标明确

边界清楚

能停得住的小 Agent

## 四 最适合入门的 Agent 往往是单一任务型

不是所有 Agent

一开始都要做成万能助手

对普通学习者来说

最适合入门的

往往是单一任务型 Agent

也就是它只负责一类事情

比如

- 帮你整理一批资料
- 帮你检查一篇文稿的问题
- 帮你跑一轮固定的代码检查
- 帮你把一个需求先整理成第一版任务拆解

这种 Agent 的好处是

目标清楚

验证容易

你也更容易判断

它到底帮上忙了没有

如果一开始就想做

什么都能做的 Agent

最后通常会变成

什么都沾一点

但没有一个地方特别稳

## 五 你可以把 Agent 开发理解成工作流再往前走一步

如果你已经学过前面的个人 AI 工作流

那理解 Agent 开发会更容易

因为它们不是完全两回事

工作流更像是

你把步骤设计出来

你自己在推进

AI 参与其中一些节点

Agent 则更像是

你把目标 步骤规则 工具边界设计好

然后让 AI 在这个框架里

替你推进更多具体动作

所以 Agent 开发

本质上不是突然进入一个全新世界

而是把原来靠人手动推进的部分

进一步交给 AI 代理去执行

## 六 这一节你先拿走这个最小思考模板

如果你想判断

一件事能不能做成一个简单 Agent

可以先用下面这个模板来想

```text
我要做的任务是什么

这个任务的目标是什么

这个任务需要哪些输入材料

它最适合调用哪些工具

哪些步骤可以自动推进

哪些步骤必须让我确认

如果失败了 它应该在哪里停下来

最后它应该交给我什么结果
```

你能把这几件事想清楚

就已经比一上来就找框架稳很多了

## 七 这一节你先记住这句话

如果要把这一节收成一句话

我会这样说

```text
Agent 开发不是先追求一个什么都能做的超级助手
而是先把目标 工具 边界 和推进方式设计清楚
做一个小而稳的任务代理
```

到这里

第九章就从

怎么和 AI 一起做项目

一路讲到了

怎么理解更进阶的提示词工程

Harness

以及 Agent 开发
