# Vibe Coding 适合做什么 不适合做什么

## 这节课的目标

学会先判断

一件事适不适合用 Vibe Coding 来做

很多人开始用 AI 做项目

最容易犯的错误

不是不会下指令

而是一上来就选错题

明明只适合先做一个小工具

结果上来就想做完整系统

明明只是想验证一个小想法

结果一开始就把权限 数据库 支付 部署都拉进来

最后不是 AI 不行

而是任务本身就不适合这么开局

所以这一节不讲怎么写需求

也不讲怎么生成第一版

我们先解决一个更前面的问题

什么任务适合用 Vibe Coding

什么任务现在先别用

## 一 先记住一个基本判断

Vibe Coding 最适合做的

不是那种要求一次就做对的大项目

而是可以快速做出第一版

快速看到结果

再继续修改的小项目

所以你判断一件事适不适合

最简单可以先看 3 个问题

- 能不能先做出一个最小版本
- 做出来之后能不能很快验证
- 就算第一版不完美 会不会造成太大代价

如果这 3 个问题里

大部分答案都是能

那它通常就比较适合

如果这 3 个问题里

大部分答案都是不能

那就说明这件事更适合先做传统规划

或者先找更稳妥的方式

## 二 什么样的任务通常适合

第一类

是小而完整的工具

比如

- 一个待办清单
- 一个表单收集页
- 一个简单的数据整理工具
- 一个个人使用的小网页
- 一个把重复动作自动化的小脚本

这些任务的共同点是

边界比较清楚

功能不多

做出来以后

你马上就能点一点 看一看

知道它是不是跑通了

第二类

是原型和演示版本

比如你有一个想法

想先做个可点击版本

给同事看

给朋友试

或者只是先验证这个流程顺不顺

这时候 Vibe Coding 很合适

因为你的目标不是一步到位

而是先把抽象想法变成一个能看到的东西

第三类

是个人场景的小需求

比如

- 个人资料整理页面
- 自己用的笔记处理工具
- 一个简单的内容生成后台
- 一个帮助自己提高效率的小助手

这类需求的好处是

用户就是你自己

反馈也最快

你今天做

今天就能试

发现不好用

马上就能改

第四类

是低风险的试验型项目

也就是就算第一版做得一般

损失也不大

最多只是重做

不会直接影响正式业务

不会碰到大量真实用户

也不会因为一个小错误

立刻带来严重后果

这种项目特别适合拿来练 Vibe Coding

## 三 为什么这些任务适合

因为 Vibe Coding 的优势

本来就不在于一次把复杂系统做完

而在于

把启动成本降下来

把试错速度提上去

让你先做出来

再通过结果继续修

所以越是这类

能快速开始

能快速看结果

能快速继续改的任务

越能发挥它的价值

你可以把它理解成

Vibe Coding 最擅长的是

把想法先推到能看见的状态

而不是直接替你兜住所有复杂度

## 四 什么样的任务先不适合

第一类

是高风险任务

比如涉及

- 支付
- 权限
- 用户隐私
- 财务数据
- 医疗 法务 合规

这些场景不是绝对不能用 AI

而是不能用 Vibe Coding 那种

先做出来再慢慢看的方式去硬冲

因为一旦出错

代价往往很高

第二类

是依赖关系特别复杂的大系统

比如一个项目同时牵涉到

前端

后端

数据库

权限系统

第三方接口

历史兼容

多人协作

这类任务不是不能让 AI 参与

但它更需要先有清楚的架构和边界

否则你一边生成

一边会把复杂度越堆越高

到后面很容易失控

第三类

是你根本没法验证结果的任务

比如你自己也不知道

一个结果对不对

好不好

符不符合业务规则

这种时候就算 AI 给你生成出来了

你也没有能力验收

那这个过程表面上很快

实际上风险很大

第四类

是目标还非常模糊的任务

比如你只是觉得

我也想做个厉害的产品

或者

我想做个像某某那样的平台

但具体给谁用

先做什么

不做什么

都还没定

这时候不要急着进入 Vibe Coding

应该先把目标收窄

不然 AI 只会把你的模糊放大

## 五 一个很实用的判断方法

如果你拿不准

可以先让自己回答下面这 5 个问题

```text
1 这个项目的第一版能不能在很小范围内完成
2 做出来之后 我能不能马上试用
3 如果第一版有问题 代价大不大
4 这个项目是不是主要为了验证想法或解决小需求
5 我自己能不能判断结果是不是基本可用
```

如果这 5 个问题里

有 4 个左右都能答是

那通常比较适合

如果很多都答不上来

那就先别急着做

先去缩小范围

或者先把任务重新定义

你也可以直接让 AI 帮你判断

```text
我想做一个【项目名称】

目前我的想法是
【写出你的项目描述】

请帮我判断
这个项目是否适合用 Vibe Coding 的方式开始

请从下面几个角度分析
1 第一版是否容易收窄
2 是否容易快速验证
3 风险是否可控
4 哪些部分适合先做
5 哪些部分现在不建议一开始就做
6 最后请给出建议
适合直接开始
还是应该先缩小范围
```

这类提问的重点不是让 AI 替你决策

而是先帮你把风险和边界看清楚

## 六 这一节你先拿走这个标准

如果要把这一节收成一句话

我会这样说

```text
Vibe Coding 最适合做
边界清楚 风险可控 能快速验证的小项目
不适合一上来就做高风险 高复杂度 又很难验收的大任务
```

所以后面下一节

我们才会继续讲

怎么把一个需求真正交给 AI

因为只有当任务选对了

后面的需求表达 生成第一版 调试和发布

才会顺起来
